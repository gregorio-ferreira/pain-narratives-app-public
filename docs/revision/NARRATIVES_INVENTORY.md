# Narratives Inventory — Revision Experiments Scope

**Companion CSV:** [`narratives_inventory.csv`](narratives_inventory.csv) — one row per non-empty
narrative_id, with size metadata (char / word / token counts) and provenance flags
(ACM baseline membership, human validation status). Generated by
[`scripts/dev/build_narratives_inventory.py`](../../scripts/dev/build_narratives_inventory.py).

The CSV deliberately excludes the narrative *text*; only counts, hashes, and flags.

---

## 1 — Headline numbers

| Set | Count | What it represents |
|---|---|---|
| Rows in `pain_narratives_app.narratives` | 303 | Total rows, including duplicates from batch repetitions |
| Empty narratives | 11 | Whitespace-only rows across 7 owners (data quality issue, see §5) |
| Non-empty rows | **292** | One per (narrative_id) — what the CSV contains |
| Unique narratives by `narrative_hash` | **177** | Deduplicated by content |
| **Used in published GPT-5 ACM baseline** (groups 38, 39, 40) | **152** | Three repetitions × 152 narratives = 456 synthetic experiments |
| Have any human validation (assessment or questionnaire feedback) | 47 | Across 19 expert evaluators |
| **Human-comparable: in ACM baseline AND have human feedback** | **40** | The subset on which LLM-vs-human correlation analyses are computed |

These are by `narrative_hash` (content-deduplicated). The published synthetic batch
and the human evaluations use **different `narrative_id`s of the same content**,
which is why a per-id intersection looks empty (0) while the per-content
intersection is 40. The CSV exposes both views.

## 2 — Provenance: how each set was identified

The published GPT-5 baseline corresponds to three experiment groups in the existing schema:

| Group ID | Description | Created | Narratives | Repetitions |
|---|---|---|---|---|
| 38 | Batch Run 2 - ACM Publication | 2026-01-11 | 152 | 1 |
| 39 | Batch Run 3 - ACM Publication | 2026-01-11 | 152 | 1 |
| 40 | Batch Run 4 - ACM Publication | 2026-01-13 | 152 | 1 |

(Earlier groups 35 + 36 represent Run 1 — 145 + 7 = 152 — split because 7 narratives failed
in the initial run and were retried. The three groups above cover Runs 2–4, which is what
`scripts/run_batch_evaluation.py --from-groups 38,39,40` extracts.)

The 152 narrative_ids referenced by these groups define the **canonical input set
for the revision experiments**.

Human validation lives in two tables:

- `pain_narratives_app.assessment_feedback` — 56 unique narrative_ids, 19 evaluators
- `pain_narratives_app.questionnaire_feedback` — 51 unique narrative_ids, 18 evaluators

These were produced via the Streamlit expert UI in groups 14–32 ("Evaluacion_*"
groups, October 2025). Aggregated by content hash, **47 distinct narratives** have at
least one human evaluation.

## 3 — Token-count distribution (gpt-5 / gpt-4o tokenizer, `o200k_base`)

| Subset | n | min | median | mean | max |
|---|---:|---:|---:|---:|---:|
| All non-empty rows | 292 | 1 | 271 | 507.8 | 12,806 |
| ACM baseline only | 152 | 4 | 304 | 489.4 | 3,718 |
| Human-comparable subset | 40 | 38 | 272 | 459.3 | 2,161 |

The 12,806-token outlier is narrative 302/303, a Catalan research grant
application accidentally pasted into the system (see §5). It's not in the
revision input set.

The CSV also records `tokens_cl100k` (gpt-3.5 / gpt-4 tokenizer) for legacy
reference. Anthropic, Amazon Nova, DeepSeek and Qwen don't have a public Python
tokenizer comparable to tiktoken; their token counts will be observed at
inference time and persisted in `request_response.usage`. As a planning proxy,
`tokens_o200k × 1.05–1.15` is a reasonable estimate for non-OpenAI models.

## 4 — Implications for the revision matrix

### 4.1 Use the same 152 narratives as the published baseline

To make the revision a clean apples-to-apples extension, the new
(model × temperature × prompt) sweep runs over the 152 narrative_ids in groups
38, 39, 40. The existing `--from-groups 38,39,40` flag in
`scripts/run_batch_evaluation.py` already does this extraction.

All revision cells will therefore have a 1:1 narrative_id → narrative_id link
back to the published GPT-5 results, even if the content overlaps with human
evaluations under different narrative_ids elsewhere in the table.

### 4.2 Human-vs-model correlation lives on the 40-narrative subset

The `hash_human_comparable` flag in the CSV marks the 40 narratives where:

- the published GPT-5 baseline was run, *and*
- a human (or multiple humans) provided assessment / questionnaire feedback

Per-cell Pearson r and RMSE vs human ground truth should be computed only on
this subset, exactly as the original paper did. The remaining 112 narratives
in the ACM baseline are synthetic-only.

### 4.3 Cost-estimate refinement

Plan §5.2 of `REVISION_EXPERIMENTS_PLAN.md` assumed ~350 input tokens median per
narrative; the empirical median over the 152 ACM baseline narratives is **304**
(slightly lower) but the **mean is 489** (long-tailed). Cost estimates should be
recomputed with empirical mean × the per-call full prompt overhead. The `tokens_o200k`
column in the CSV is the input for that calculation.

## 5 — Data quality findings (pre-existing, not introduced by the revision work)

Three issues to be aware of; none affect the 152 ACM baseline:

| Issue | Affected rows | Notes |
|---|---|---|
| Empty narratives (whitespace-only, `char_count = 0`) | 11 (rows 38, 39, 48, 50, 51, 57, 79, 80, 81, 121, 134) | Across 7 distinct owners. All hash to `e3b0c44…` (SHA-256 of empty string). Excluded from the inventory CSV (`WHERE char_count > 0`). |
| Non-narrative content pasted in | 2 (rows 302, 303) | A 53 kB Catalan research grant application annex. Owner_id 4. Not in any experiment group. |
| Orphan narratives (in no reported experiment, no human validation) | 20 (by hash) | Likely from earlier pilot runs. The CSV's `hash_in_acm = False AND hash_has_any_human = False` filter surfaces them. |

The original published Table-N counts of "152 narratives" exclude these by
construction — they were never in groups 35–40.

## 6 — Reproducing the inventory

```bash
uv run python scripts/dev/build_narratives_inventory.py
```

Reads the live `pain_narratives_app` schema, computes token counts via
`tiktoken.get_encoding("o200k_base")` and `cl100k_base`, joins membership
flags, and writes `docs/revision/narratives_inventory.csv`. Re-run after any
new batch group lands to refresh the "ACM baseline" definition (the
`ACM_BATCH_GROUP_IDS` constant at the top of the script controls which groups
count as "baseline").
